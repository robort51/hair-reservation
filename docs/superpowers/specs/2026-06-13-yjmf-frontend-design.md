# YJMF 前端设计方案

## 目标

为单门店美发预约系统增加一个 Next.js 前端项目，包含客户预约端和店内管理端。客户预约端面向顾客，强调品牌感和预约体验；店内管理端面向老板或店员，强调信息密度、操作效率和状态清晰。

店名为 `YJMF`。第一版不做复杂登录、支付、短信、会员、优惠券和图片上传，先完成“顾客能顺畅预约，店里能查看和处理预约”的可用闭环。

## 前端项目结构

前端项目放在仓库根目录的 `web/`：

```text
web/
  app/
    page.tsx
    book/page.tsx
    admin/page.tsx
    admin/appointments/page.tsx
  components/
    brand/
    booking/
    admin/
    ui/
  lib/
    api.ts
    format.ts
    types.ts
```

页面入口：

```text
/                  YJMF 品牌首页
/book              客户预约流程
/admin             今日预约看板
/admin/appointments 预约列表和处理
```

## 技术选择

- 使用 Next.js、React、TypeScript。
- 使用 CSS Modules 或全局 CSS 变量建立设计系统；第一版避免引入大型 UI 框架。
- API 地址通过环境变量配置：`NEXT_PUBLIC_API_BASE_URL=http://localhost:3000`。
- 前端只消费现有 NestJS API，不直接访问数据库。
- 第一版后台不做登录鉴权，后续上线前再补身份校验。

## 视觉方向

整体方向是“精致沙龙编辑感 + 安静高效工作台”，不是简单白底表单。

客户预约端：

- 第一眼必须记住 `YJMF`，首页用大字号品牌字标、细线网格、服务时间和预约入口形成第一屏。
- 色彩以象牙白、深墨黑、暖铜金为主体，辅以雾绿色表示可预约或成功状态。
- 页面有层次：背景细纹、分区线、服务卡片、步骤进度、时间块和底部确认栏，而不是纯列表。
- 动效克制但存在：进入页面时品牌、服务卡片、时间块分层出现；按钮和时间选择有细微位移和阴影变化。

店内管理端：

- 更像专业工作台，低装饰、高可读性。
- 采用深色侧栏或顶部栏，主内容为浅色工作区；状态色清晰区分待完成、已完成、已取消。
- 卡片半径保持 8px 以内，避免圆润玩具感。
- 操作按钮使用图标加短文字，关键操作如完成、取消、查看详情需要一眼可辨。

## 客户预约端流程

### 首页 `/`

首页不是营销长页，只做预约前的品牌入口。

首屏内容：

- 品牌名：`YJMF`
- 主标题：`预约你的专属发型时间`
- 辅助信息：营业时段、服务范围、单店预约
- 主要按钮：`立即预约`
- 次要入口：`查看服务`

首页下方展示三个信息区：

- 服务项目：洗吹、洗剪吹、基础护理、染发、烫发
- 预约方式：选择服务、员工、时间后提交
- 到店提示：按预约时间到店，修改预约可联系店内

### 预约页 `/book`

预约页采用手机优先的步骤式流程，桌面端居中展示为“预约卡片 + 摘要侧栏”。

步骤：

1. 选择服务
   - 请求 `/service-items`。
   - 按分类展示服务名称、时长、价格。
   - 服务卡片显示价格、预计耗时和简短说明。

2. 选择员工
   - 请求 `/staff`。
   - 展示员工姓名、头衔、可服务项目。
   - 第一版允许“任意员工”作为快捷选项，但提交预约时仍需要明确员工；如果选任意员工，前端在可预约时间里按员工分组展示。

3. 选择日期和时间
   - 请求 `/availability?serviceItemId=...&date=...`，如果已选员工则带上 `staffId`。
   - 日期横向滚动显示未来 7 天。
   - 时间块区分可选、已满、当前选中。
   - 无可预约时间时给出清晰空状态。

4. 填写信息
   - 输入姓名、手机号、备注。
   - 底部固定确认栏显示服务、员工、时间、价格。

5. 提交结果
   - 调用 `POST /appointments`。
   - 成功后显示预约编号、服务、员工、到店时间。
   - 失败时显示后端错误信息，冲突时提示用户重新选择时间。

## 店内管理端流程

### 今日预约看板 `/admin`

这是店内日常默认入口。

内容：

- 今日预约总数
- 待完成数量
- 最近一小时预约
- 预约时间线
- 快捷操作：完成、取消、查看详情

预约卡片显示：

- 时间段
- 顾客姓名和手机号
- 服务项目
- 员工
- 状态
- 备注

操作行为：

- 完成预约：调用 `PATCH /appointments/:id/complete`。
- 取消预约：调用 `PATCH /appointments/:id/cancel`，第一版使用简单弹窗输入取消原因。

### 预约列表 `/admin/appointments`

列表用于查看全部预约。

第一版筛选：

- 日期
- 状态
- 员工

列表字段：

- 开始时间
- 顾客
- 服务
- 员工
- 状态
- 操作

## 后续管理页面

第一版可以先保留入口或空状态，不阻塞预约闭环。

后续补齐：

- `/admin/services`：服务分类和服务项目维护。
- `/admin/staff`：员工信息和员工可服务项目维护。
- `/admin/schedules`：每周排班和不可预约时间维护。

## API 数据流

客户预约端：

```text
GET /service-items
GET /staff
GET /availability
POST /appointments
```

管理端：

```text
GET /appointments
PATCH /appointments/:id/complete
PATCH /appointments/:id/cancel
```

所有接口使用后端统一响应结构：

```ts
type ApiResponse<T> = {
  data: T | null;
  error: {
    code: string;
    message: string;
  } | null;
};
```

前端 `lib/api.ts` 统一处理：

- 拼接 API 地址。
- 解析统一响应结构。
- 把 `error.message` 转为页面错误提示。
- 网络失败时显示“网络异常，请稍后重试”。

## 状态和错误处理

- 加载中：服务卡、员工卡和时间块使用骨架态。
- 空数据：预约时间为空时给出重选日期和员工的提示。
- 冲突：预约提交返回 `APPOINTMENT_CONFLICT` 时，保留已选服务和员工，只要求用户重新选择时间。
- 表单错误：手机号和姓名在前端做基础校验，最终以后端校验结果为准。
- 后台操作失败：保留当前列表状态，顶部显示错误提示。

## 响应式设计

客户预约端：

- 手机端为主，底部固定确认栏。
- 桌面端使用居中预约面板，右侧显示预约摘要。

管理端：

- 桌面端为主，适合电脑和平板。
- 窄屏下侧栏变为顶部导航，预约卡片改为单列。

## 第一阶段实现范围

第一阶段前端只做这些页面：

- `YJMF` 首页。
- 客户预约完整流程。
- 后台今日预约看板。
- 后台预约列表，支持完成和取消。

不做：

- 登录权限。
- 支付。
- 短信通知。
- 小程序代码。
- 复杂统计报表。
- 图片上传和作品集。

## 验收标准

- 顾客可以从首页进入预约流程并成功创建预约。
- 同一员工同一时间段重复预约时，前端展示冲突提示。
- 店内可以看到预约列表，并完成或取消预约。
- 页面在手机宽度下不出现文字遮挡、按钮溢出、步骤跳动。
- 管理端在桌面宽度下能快速扫读今日预约。
- 前端 README 明确说明本地启动方式和 API 地址配置。
